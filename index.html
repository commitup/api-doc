    Introduction

    Welcome to the Money Transfer API reference.

    This API serves as the primary gateway to facilitate money transfers through PayPorter platform.

    The Money Transfer API is organized according to REST principles

    Base URL : https://online-mig.payporter.com.tr:8586/online

    General API Information

    Access Token

    All endpoints require a Bearer token in the Authorization header. You can get the token by calling the /login endpoint with your credentials. You should keep this token until it expires. When the token expires, you will need to generate a new one. Please do not call login before each request. Otherwise, you will get HTTP 429 Too Many Requests response.

    Response and Errors

    If request is success you will get HTTP 200 OK response. If request is failed you will get HTTP 4XX response.

    For business errors, you will get HTTP 406 response with success field as false in response body. MessageCode field will be filled with error code. You need to check messageCode which is uniqueue.

    HTTP 500 Internal Server Error response, It means that there is a problem with the server. Please check your transfer or payment status before retrying the request.

    LOGOUT in messageCode field from any endpoint, you should call login endpoint again to get a new token. HTTP 401 Unauthorized or HTTP 406 response may be received for LOGOUT.

    code field in response is not unique. Its kept for legacy integrations. You should check messageCode field.

    In this API, response structure is like : 

    "body": {
    "responseObject": {
     .....
    },
    "restHeader": {
      "code": "string",
      "message": "string",
      "success": true,
      "messageCode":"string"
    }
}

    Rate Limiter

    This API has a rate limiter. If you exceed limits, you will get HTTP 429 Too Many Requests response.

    You need to fill externalfirm-user-code in request header. If it is not provided, please ask for.

    Transfer

    Transfer is the main resource of the API. It represents a money transfer request. It is created by the sender and then it is processed by the PayPorter platform. Validation is needed to be done before the transfer is created. After validation is done, confirm is need to be called to create the transfer.

    In validations reponse header, you will get and operation-id. This operation-id is needed to be used in confirm request.

    Here are the key points about the API and its transfer types:

    • The API allows for money transfers to be made to other people.
    • There are three types of transfer: by name, bank account, or electronic wallet.
    • The recipient is identified using their name, bank account details, or wallet details, respectively.
    • Before a transfer is created, it must be validated.
    • The confirm method must be called to create the transfer after it has been validated.
    • In the response header of the validate method, you will receive an operation-id which is needed for the confirm request.

    Here are the steps involved in using the money transfer API:

    1. The sender creates a transfer request using the POST method and the appropriate endpoint for the transfer type (e.g. /mt-api/V2/moneysend/to-name/validate for a transfer to name).
    2. The sender includes the necessary parameters in the request body, including the Person objects for the sender and recipient, as well as the transfer amount and other details.
    3. The sender includes the authentication token in the request header (e.g. Authorization: Bearer {auth-token}).
    4. The API processes the request and returns a response indicating whether the request was successful or not.
    5. If the request was successful, the API will return an operation-id in the response header.
    6. The sender uses the operation-id to make a confirm request using the POST method and the appropriate endpoint (e.g. /mt-api/V2/moneysend/confirm).
    7. The sender includes the operation-id in the request header (e.g. operation-id: abc123).
    8. The API processes the confirm request and creates the transfer

    Payment

    Payment is the process for incoming transfer with a reference number. Validation is needed to be called before the payment is confirmed.

    In validations reponse header, you will get and operation-id. This operation-id is needed to be used in confirm request.

    Here are the key points about the API and its transfer types:

    • The API allows for cash pickup transfer to be collected by receiver.
    • A payment is processed with a reference number from external/remmittance firm
    • The confirm method has to be called to pay transfer after it has been validated.
    • In the response header of the validate method, you will receive an operation-id which is needed for the confirm request.

    To track the status of an EFT transfer, you can periodically query the transaction's status until it reaches the Final Status or COMPLETED status.

    There are two methods you can use to check the status of a transfer:

    • Use the POST /eft-api/V2/transfer/get-transfer-list endpoint with an order date range to retrieve a list of all your orders.

    • Query the status of a specific transfer using the following endpoints:

      • GET /eft-api/V2/transfer/check-status-by-ext-firm-id/{extFirmRefId}
      • GET /eft-api/V2/transfer/check-status-by-transfer-order-ref/{transferOrderRefId}

    ⚠️ Rate Limiter! Please note that if you choose to query the status of each transfer individually, there is a rate limiter in place to prevent more than 60 queries per minute.

    Once a transfer reaches the COMPLETED, REJECTED, REFUND, or CANCEL status, you should stop querying its status (if you are using the specific transfer based query methods).

    Since a REFUND can occur at any time after a transfer is COMPLETED, it is recommended to call the "Get Refund List" endpoint multiple times a day.

    To retrieve the list of transactions that were refunded today, you can use the "Get Refund List" endpoint with the SYSDATE(today) parameter. You do not need to query with the original order date.

    We have a webhook for EFT status notification. If you want to use, You need to implement endpoint like below

    IMPORTANT! Webhook needs to be idempotent. We may notify the same status multiple times in some cases (timeout, system error)

    Secure Webhook Implementation

    For the secure webhook implementation, we utilize the HmacSHA256 algorithm for hashing. The output format for the hashed value is in hexadecimal (hex).

    Secret Value

    A secret value, which we will provide, is used during the hashing process. This secret value ensures the security and integrity of the webhook.

    Hashing Process

    The hash value included in the request is generated by hashing the entire request using the HmacSHA256 algorithm with the provided secret. This process ensures that the request has not been tampered with.

    Request Header

    The resulting hashed value is included in the request under the "request-sign" header.

    • If everything is OK, then we will return HTTPS status 200 and success = true.
    • If SENDER_REF_NOT_FOUND, ORDER_REF_NOT_FOUND, then return HTTPS status 404 and success = false.
    • If STATUS_NOT_VALID returns HTTPS status 400 and success = false
    • In other cases, HTTP status 500 and success=false will be returned
